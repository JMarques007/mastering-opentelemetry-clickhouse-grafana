# Course Checklist

Use this checklist before considering any module ready for review or release.

## Module Metadata

- [ ] Module folder follows the standard naming convention.
- [ ] `README.md` exists.
- [ ] `module.md` exists.
- [ ] `quiz.md` exists.
- [ ] `references.md` exists.
- [ ] `slides/README.md` exists.
- [ ] `diagrams/` exists.
- [ ] `images/` exists.
- [ ] `examples/` exists.
- [ ] `labs/` exists.

## Documentation Quality

- [ ] The module starts with a narrative introduction.
- [ ] The introduction explains why the topic matters.
- [ ] The module is written in professional English.
- [ ] The writing flows like a technical book chapter.
- [ ] The module avoids unnecessary jargon.
- [ ] The module avoids bullet-heavy sections.
- [ ] Each section has a clear purpose.
- [ ] Concepts are introduced progressively.
- [ ] The module explains why before how.
- [ ] The module contains practical examples.
- [ ] The module contains production-oriented guidance.

## Required Teaching Blocks

- [ ] At least one Production Example.
- [ ] At least one Architect Note.
- [ ] At least one Best Practice block.
- [ ] At least one Common Mistake block.
- [ ] Key Takeaways section.
- [ ] Common Interview Questions section.
- [ ] Next Module transition.

## Technical Accuracy

- [ ] Terminology is consistent with OpenTelemetry and Grafana usage.
- [ ] Official names are used correctly.
- [ ] Claims are accurate and not overstated.
- [ ] Vendor-neutral explanations are maintained where appropriate.
- [ ] ClickHouse and Grafana references are technically correct.
- [ ] Architecture diagrams match the described flow.

## Diagrams and Images

- [ ] Every diagram has a clear teaching purpose.
- [ ] Architecture diagrams have an editable source where possible.
- [ ] Mermaid is used for simple editable diagrams.
- [ ] Static images are not used as a substitute for editable architecture diagrams.
- [ ] The module does not contain too many images.
- [ ] Image filenames are meaningful.
- [ ] Diagram captions explain what learners should notice.

## Slides

- [ ] Slide outline exists.
- [ ] The deck supports approximately 1 hour of delivery.
- [ ] Each slide has one main idea.
- [ ] Slides are visual and concise.
- [ ] Slides do not duplicate the full documentation.
- [ ] Speaker notes or delivery guidance are included.
- [ ] Diagrams required for slides are identified.

## Labs and Exercises

- [ ] At least one exercise or lab exists.
- [ ] The lab has a clear learning objective.
- [ ] Prerequisites are listed.
- [ ] Steps are clear and ordered.
- [ ] Expected results are described.
- [ ] Troubleshooting guidance is included.
- [ ] Optional extensions are included where useful.

## Quiz

- [ ] Quiz questions test understanding, not memorization only.
- [ ] Answers are included.
- [ ] Questions align with the module objectives.
- [ ] At least one question asks for reasoning or trade-off analysis.

## References

- [ ] References are official where possible.
- [ ] References are relevant to the module.
- [ ] Links are grouped by topic.
- [ ] No random blog posts are used unless justified.

## Editorial Review

- [ ] No placeholder text remains.
- [ ] No repeated filler language.
- [ ] Tone is consistent with the Course Bible.
- [ ] Section transitions are natural.
- [ ] The module feels useful for real training.

## Release Readiness

A module may be marked as `First review draft` only when all required files exist and the documentation is usable for review.

A module may be marked as `Ready for training` only when documentation, slides, labs, diagrams, quiz and references have been reviewed.

A module may be marked as `Released` only after technical, editorial and lab validation.
