# Style Guide

This document defines the editorial and visual standards for the course.

## Writing Style

- Use clear professional English.
- Prefer active voice.
- Explain **why** before explaining **how**.
- Avoid unnecessary jargon.
- Use real-world examples when they improve understanding.
- Avoid long bullet-only sections; the documentation should read like a professional manual.

## Module Structure

Each module should follow this structure:

1. Overview
2. Learning Objectives
3. Prerequisites
4. Main Content
5. Best Practices
6. Common Mistakes
7. Practical Exercise or Lab
8. Summary
9. Quiz
10. References
11. Next Module

## Slides

- Use 16:9 format.
- Use minimal text.
- Prefer diagrams over bullet-heavy slides.
- Maximum five bullets per slide.
- Slides support the presenter; the documentation contains the detail.

## Diagrams

- Diagrams should be editable whenever possible.
- Prefer PowerPoint or Draw.io as the source format.
- Export diagrams to SVG and PNG for documentation.
- Use white or very light backgrounds.
- Keep architecture diagrams readable and uncluttered.

## Images

- Use images sparingly.
- Use them mainly as hero images or visual separators.
- Do not use generated images to explain technical architecture when an editable diagram is more appropriate.

## Terminology

Use consistent terminology:

- OpenTelemetry Collector
- OpenTelemetry SDK
- OTLP
- Resource Attributes
- Span Attributes
- Traces, Metrics and Logs
- Observability Backend

Avoid inconsistent alternatives such as "resource tags" or "OTel DB".
